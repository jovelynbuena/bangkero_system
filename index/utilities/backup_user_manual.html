1. Page Overview & Summary Cards

The Backup & Restore page lets admins protect the system by saving snapshots of the database or the entire project, and reverting to those snapshots when needed.

–Total Backups (SQL + ZIP combined)
Last BackupDate and time of the most recent backup file

The two summary cards at the top of the page give a quick status snapshot so you know at a glance how many backups exist and when the last one was taken.

The Auto Backup Status card is temporarily hidden on the current version. It will return when the scheduled-backup feature is activated in System Settings.

2. Create Database Backup

Generates a .sql file that contains a complete snapshot of all tables, structures, and data in the current database.

Button

Create Backup Now

How it works

1
Click Create Backup Now.
2
The system queries every table (SHOW TABLES) and writes a DROP TABLE IF EXISTS + CREATE TABLE + INSERT block for each one.
3
The resulting file is saved as backup_YYYY-MM-DD_HH-II-SS.sql inside /index/utilities/backups/.
4
A green success banner appears with a Download button so you can save the file locally right away.
5
The activity log records who created the backup and when.
If the storage limit is reached, the Create Backup Now button will be disabled. Delete older backups first to free space.

Output file

  • Format: .sql (plain-text SQL statements)
  • Includes: all tables + all data rows
  • Can be opened in phpMyAdmin, MySQL Workbench, or any text editor

3. Restore Database Backup (Upload .sql)

Lets you upload any previously downloaded .sql backup file and overwrite the current database with it.

File field

Click Select SQL file and choose a .sql backup file from your computer. Only .sql files are accepted.

Button

Restore Database

How it works

1
Select your .sql file and click Restore Database.
2
A SweetAlert confirmation dialog appears — read it carefully, then confirm.
3
The system parses the SQL into individual statements and executes each one. FOREIGN_KEY_CHECKS is disabled during restore to avoid constraint errors.
4
A success or partial-warning toast message is shown with the count of successful and failed queries.
5
The activity log records the restore action.
Destructive action! Restoring a database completely overwrites the current data. All records created after the backup date will be lost. Always create a fresh backup before restoring.

4. Create System Backup (.zip)

Creates a single .zip archive containing both the database dump (as database_dump.sql) and the project files. Use this for a complete server migration or disaster-recovery backup.

Button

Create System Backup

What is included in the ZIP

  • database_dump.sql – full database export
  • index/ – all PHP pages and partials
  • uploads/ – all uploaded images/files
  • config/ – configuration files
  • css/ – stylesheets
  • images/ – static images
  • composer.json, composer.lock

What is excluded

  • vendor/ – PHP dependencies (re-run composer install after restore)
  • index/utilities/backups/ – previous backup files
  • node_modules/ – front-end dependencies

How it works

1
Click Create System Backup — a confirmation dialog appears.
2
A background form is submitted. A progress dialog keeps you informed while the ZIP is being built (this can take a minute for large file sets).
3
The ZIP is saved as system_backup_YYYY-MM-DD_HH-II-SS.zip in the /backups/ folder.
4
A success toast with a Download System Backup button appears so you can save the ZIP locally.
System backups can be very large (tens of MB to hundreds of MB) depending on how many uploads exist. Ensure your server has enough free disk space and the PHP ZipArchive extension is enabled.

5. Restore System Backup (Upload .zip)

Upload a previously downloaded system_backup_*.zip file to restore both the database and all project files at once.

File field

Click Select ZIP file and choose a .zip system backup from your computer. Only files ending in .zip that contain a database_dump.sql inside are accepted.

Button

Restore System

How it works

1
Select the ZIP and click Restore System.
2
A strong warning dialog confirms your intent — type "RESTORE" or click Confirm to proceed.
3
All files in the ZIP (except database_dump.sql) are extracted and overwrite the existing project files.
4
database_dump.sql is parsed and each SQL statement is executed against the live database.
5
A result toast shows the number of files extracted and SQL queries executed, plus any warnings.
Highly destructive! This operation replaces both the database and all project files. Use only if you need to fully revert the system. Create a fresh backup before proceeding.
After a system restore you may need to run composer install in the project root to regenerate the vendor/ folder, since it is excluded from the backup.

6. Backup History Tabs

Below the action cards is the Backup History section with two tabs:

SQL Backup Tab

Lists all .sql database-only backup files. Each row shows the filename, date created, file size, and action buttons.

System Backup Tab

Lists all .zip system backup files. Same layout as the SQL tab. The badge next to each tab label shows the count of files.

Both tabs support pagination (10 items per page) and live filtering — see Section 8.

The row counter at the bottom of each tab shows "Showing X – Y of Z" so you always know where you are in the list.

7. Per-Backup Actions

Each backup row in the history list has up to three action buttons:

Download

Download

Saves the backup file to your computer. Use this to keep an off-site copy of your backups for extra safety.

Restore (from history)

Restore

Restores the database (SQL tab) or system (System tab) directly from the server-side file — no upload needed. A confirmation dialog appears before anything changes.

  • SQL tab: restores database only.
  • System tab: restores files + database (same as uploading the ZIP manually).

Delete

Delete

Permanently removes the backup file from the server. A SweetAlert confirmation dialog prevents accidental deletion. This action cannot be undone.

Only .sql files can be deleted via the Delete button. System ZIP files use the same button in the System Backup tab. Always keep at least one off-site copy before deleting server-side backups.

8. Search & Filter Backups

Both the SQL and System backup tabs have identical filter toolbars above the list:

Filter ControlWhat it does
Search boxType any part of the filename to filter the list in real time.
Sort dropdownOrder by: Newest First (default), Oldest First, Largest, Smallest.
Date From / Date ToShow only backups created within a date range.
Min Size / Max Size (MB)Show only files within a size range (enter values in megabytes).

Filters apply separately to the SQL tab and the System tab. Changing filters on one tab does not affect the other.

10. Tips & Best Practices

  • Before any major change (importing data, updating records in bulk, upgrading the system), always create a fresh database backup first.
  • After creating a backup, click Download immediately and save it to an external drive or cloud storage — server storage is not a substitute for off-site backups.
  • System backups vs Database backups: Use a System Backup when migrating to a new server. Use a Database Backup for routine data protection — it is faster and smaller.
  • Partial restore warnings: If a restore completes with warnings (yellow banner), some queries failed — this usually happens with minor version differences in MySQL. Check the warning details and verify your data.
  • Storage: Old backups accumulate quickly. Clean up files older than 30 days if you have regular backups in place.
  • Restore after system restore: After restoring a System Backup ZIP, run composer install on the server to regenerate the vendor/ folder.
  • CSRF protection: All backup/restore forms include a CSRF token. If you see an "Invalid session token" error, reload the page and try again.
Good habit: Create a database backup every week and a system backup every month, or before and after every major system update.
